Battleship Project Overview
==========================

This document explains how the Battleship application works, including the runtime flow, domain model, API behavior, persistence, and test coverage.

1. Project Structure
--------------------

- Battleship.csproj: main application project using ASP.NET Core.
- Program.cs: application startup, DI configuration, and routing.
- Controllers/: API endpoints for games and completed summaries.
- Data/: Entity Framework Core DB context.
- Domain/: game logic, board model, ships, shot handling, randomization interface.
- Dtos/: request/response contracts for the REST API.
- Models/: persistent game summary entity.
- Battleship.Tests/: separate xUnit test project.

2. Startup and Dependency Injection
-----------------------------------

Program.cs does the following:
- Creates the WebApplication builder.
- Registers controllers and Swagger/OpenAPI generation.
- Registers BattleshipContext with SQLite using a connection string from configuration or default file "battleship.db".
- Registers a singleton IRandomProvider implementation (RandomProvider) for ship placement.
- Builds the app.
- Ensures the SQLite database exists by calling db.Database.EnsureCreated() inside a service scope.
- Enables Swagger UI in development.
- Serves static files and maps controllers.
- Falls back to index.html for client-side routing.
- Runs the app.

3. Domain Model
---------------

The core game logic lives in Domain/.

Game.cs
- Represents one active game.
- Constructor takes a game ID, board size, and IRandomProvider.
- Creates a new Board instance.
- Exposes Id, Board, CreatedAt, and IsWon.
- IsWon is true when Board.ShipsRemaining == 0.

Board.cs
- Represents the game board state and fleet placement.
- Contains board size, shot cache, and ship list.
- Places the fleet in the constructor using PlaceFleet().
- Provides IsInBounds(), Fire(), ShotsFired, ShipsRemaining, Ships, and Shots.

Ship.cs
- Represents a single ship.
- Stores name, length, positions, and hit positions.
- Can report whether it occupies a position and whether it is sunk.
- RegisterHit(position) records a hit if the ship occupies that position.

BoardPosition.cs
- Immutable struct with X and Y coordinates.

ShotResult.cs
- Immutable record containing ShotOutcome and optional ShipName.
- ShotOutcome is an enum: Miss, Hit, Sunk.

IRandomProvider.cs
- Interface abstracting random number generation.
- Allows deterministic testing by replacing the random provider.

4. Ship Placement Algorithm
---------------------------

Board.PlaceFleet():
- Defines 5 ships with lengths: Carrier 5, Battleship 4, Cruiser 3, Submarine 3, Destroyer 2.
- For each ship, it tries up to 1000 placement attempts.
- Chooses horizontal or vertical using Random.Next(2).
- Chooses start X/Y based on board size minus ship length.
- Builds the requested positions and checks overlap with existing ships using IsOccupied.
- If placement collides, it retries.
- If no placement is found after 1000 attempts, it throws InvalidOperationException.

Placement rules:
- Ships must stay in bounds.
- Ships must not overlap.
- RandomProvider controls randomness.

5. Shot Resolution and Game Progress
------------------------------------

Board.Fire(position):
- Throws ArgumentOutOfRangeException if position is out of bounds.
- If the shot was already taken, returns the cached ShotResult.
- If no ship occupies the position, records and returns Miss.
- If a ship occupies the position, calls ship.RegisterHit(position).
- If the ship becomes sunk after the hit, returns Sunk with ship name.
- Otherwise returns Hit with ship name.
- Stores the result in the shot cache.

A shot does not remove or change ship positions other than marking hits.

Game.IsWon becomes true once all ships are sunk.

6. API Flow
-----------

GamesController handles all active-game endpoints.

Active game storage:
- Uses a static Dictionary<string, Game> named ActiveGames.
- Uses a static LockObject to synchronize access.
- Games are stored in memory only for the running process.

POST /games
- Query parameter: boardSize (default 10).
- Validates boardSize between 5 and 20.
- Creates a GUID game ID.
- Constructs a new Game with boardSize and the registered random provider.
- Stores it in ActiveGames.
- Returns CreatedAtAction with CreateGameResponse containing gameId and boardSize.

POST /games/{id}/shots
- Accepts FireShotRequest with X and Y.
- Validates request body.
- Looks up the game inside ActiveGames using lock.
- Returns 404 if game not found.
- Returns 400 if the game is already won.
- Validates coordinates with game.Board.IsInBounds. If out of bounds, returns 400.
- Fires the shot with game.Board.Fire(position).
- Checks if the game is now won.
- If the game just finished, calls PersistCompletedGame(game).
- Returns ShotResponse with:
  - Outcome: miss/hit/sunk
  - ShipName if a ship was hit or sunk
  - ShotsFired
  - ShipsRemaining
  - IsWon

GET /games/{id}
- Returns a GameStateResponse with:
  - GameId
  - BoardSize
  - ShotsFired
  - ShipsRemaining
  - IsWon
  - Shots history from the board cache
- Each shot history item includes X, Y, outcome, and ship name.

PersistCompletedGame(Game game)
- Checks if summary already exists in database for that game ID.
- If not, creates GameSummary with GameId, BoardSize, ShipCount, TotalShots, CompletedAt.
- Saves changes to the database.

SummariesController
- GET /summaries returns all completed game summaries ordered by CompletedAt descending.
- Uses BattleshipContext to query GameSummaries.

7. Persistence
--------------

BattleshipContext.cs
- DbContext with DbSet<GameSummary> GameSummaries.

GameSummary.cs
- Persistent entity for completed games.
- Properties: Id, GameId, BoardSize, ShipCount, TotalShots, CompletedAt.

SQLite database
- Configured via UseSqlite in Program.cs.
- Defaults to Data Source=battleship.db if no connection string is provided.
- EnsureCreated creates the database file at startup.

8. Test Project
---------------

Battleship.Tests is a separate test project.
- Uses Microsoft.NET.Test.Sdk.
- Uses xUnit.
- Uses xunit.runner.visualstudio.
- Uses Moq (although not currently required by the tests).
- Uses Microsoft.EntityFrameworkCore.InMemory for database testing.
- References the main Battleship project.

MockRandomProvider.cs
- Implements IRandomProvider with System.Random.
- Accepts an optional seed for deterministic placement.
- Used by tests to create reproducible board layouts.

Test categories:

BoardPlacementTests
- Validates all ships are placed within bounds.
- Ensures ships do not overlap.
- Ensures deterministic placement with identical seed.
- Confirms ship count and lengths.

ShotResolutionTests
- Tests a miss when shooting empty space.
- Tests a hit when shooting a ship.
- Tests sinking a ship after all its positions are shot.
- Verifies ShotsFired increments correctly.
- Verifies ships remaining decreases after sinking.
- Confirms correct outcomes for miss/hit/sunk.

EdgeCaseTests
- Validates out-of-bounds shots throw ArgumentOutOfRangeException.
- Validates duplicate shots return the same cached result.
- Verifies duplicate shots do not increment ShotsFired.
- Checks that all boundary positions can be fired without throwing.
- Tests IsInBounds behavior for valid and invalid coordinates.

GameWinConditionTests
- Confirms new games are not won initially.
- Confirms game is won after all ships are sunk.
- Validates game is not won with ships remaining.
- Tests firing a shot after win still returns a result.
- Verifies Game.CreatedAt and Game.Id behavior.

GamePersistenceTests
- Uses InMemory EF Core database context.
- Confirms GameSummary can be saved and retrieved.
- Validates completed game summary has the correct shot count.
- Verifies summary TotalShots matches Board.ShotsFired.
- Confirms multiple summaries can be saved and queried.

9. How to run
-------------

From the root workspace:
- dotnet restore
- dotnet build
- dotnet test Battleship.Tests/Battleship.Tests.csproj

Because the test project is separate, you can also run the tests from the test folder:
- cd Battleship.Tests
- dotnet test

10. Request/Response Contracts
-----------------------------

CreateGameResponse:
- gameId: string
- boardSize: int

FireShotRequest:
- x: int
- y: int

ShotResponse:
- outcome: "miss" | "hit" | "sunk"
- shipName: string|null
- shotsFired: int
- shipsRemaining: int
- isWon: bool

GameStateResponse:
- gameId: string
- boardSize: int
- shotsFired: int
- shipsRemaining: int
- isWon: bool
- shots: array of ShotHistoryItem

ShotHistoryItem:
- x: int
- y: int
- outcome: string
- shipName: string|null

11. Important behavior details
------------------------------

- Active games are stored only in memory for the running app. If the app restarts, active games are lost.
- Completed games are stored to the database once a win occurs, but the board state is not persisted.
- Duplicate shots do not create new entries or count additional shots; they return the existing ShotResult.
- The game does not allow firing shots after the game is already finished.
- The board placement is randomized but can be made deterministic using a seeded IRandomProvider in tests.

12. Study strategy
------------------

To know the project inside and out, focus on these files in order:
- Program.cs for startup and DI.
- GamesController.cs for game flow and persistence trigger.
- Board.cs for core rules and shot resolution.
- Ship.cs for per-ship hit tracking.
- Game.cs for win detection.
- BattleshipContext.cs and GameSummary.cs for persistence.
- Tests in Battleship.Tests/Domain for expected behavior and edge cases.

This file should give you a full mental model of the application flow and the tests that prove its behavior.
